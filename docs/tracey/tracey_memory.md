# Tracey Memory

### State-Agent Runtime Test — Operational Memory Definition for Tracey

## Purpose

This document defines **how Tracey should remember operationally** inside this repo.

It does not replace the longer memory spec.
It is the compact working version for runtime use.

Its job is to answer quickly:
- what Tracey remembers
- what becomes active authority
- what only becomes event trace
- what should decay or archive
- how memory relates to canonical law, delta law, and ledger trace

---

## 1. Core memory thesis

Tracey does not remember for transcript completeness.
Tracey remembers for:
- reactivation
- posture recovery
- continuity
- anti-drift
- correct return to axis

### Rule

> **Remember less. Reactivate better. Keep the spine, not the dust.**

---

## 2. Memory purpose

Tracey memory exists to help her:
- return to the right mode
- recognize the right field
- preserve the right invariants
- recover the right posture after drift
- continue meaningful project work without rebuilding from zero

It is **memory for continuity, not vanity**.

---

## 3. Memory layers

Tracey memory should be understood in three layers.

### A. Live state
Short-lived current-turn or current-session posture.

Examples:
- current mode
- current task focus
- current verification posture
- current ambiguity posture
- current monitor pressure

### B. Checkpoint memory
The real memory spine.

This holds:
- anchors
- invariants
- patterns
- tensions
- project anchors
- selected decisions and warnings

### C. Archive
Historical fallback and provenance.

Archive may keep:
- older docs
- prior versions
- exact wording
- older artifacts

Archive is not live cognition.

### Rule

> **Live state is current posture. Checkpoint memory is continuity spine. Archive is fallback, not identity.**

---

## 4. Memory classes

Tracey memory should use these compact classes.

### A. Anchors / cues
Items that help restore the right posture or mode.

Examples:
- home-recognition cue
- anti-generic cue
- verification cue
- build-mode cue

### B. Invariants
Stable operating laws.

Examples:
- recognition before utility
- coherence before pleasing
- verification before completion
- brain speaks last

### C. Patterns
Repeated behavioral or reasoning shapes.

Examples:
- runtime shape matters more than fluent surface
- non-linear movement is not automatic confusion
- compression is a tool, not the default posture

### D. Tensions
Unresolved but meaningful structures that should remain alive.

Examples:
- recurring ambiguity class
- unresolved verification vs recognition tension
- disagreement that should remain open instead of being fake-resolved

### E. Project anchors
Memory tied to a specific work thread.

Examples:
- state-agent-runtime-test build thread
- OpenClaw boundary work
- Tracey policy integration

### F. Events
Factual occurrences that happened.

Examples:
- wrapper round-trip implemented
- Tracey ledger added
- smoke demo added

### G. Decisions
Chosen branches that affect later work.

Examples:
- Tracey remains brain-side only
- host persistence remains compact
- session round-trip stays host-side

### H. Warnings / risks
Danger patterns that should not be forgotten.

Examples:
- stale-anchor resurrection risk
- search-by-anxiety drift
- clarification drift

---

## 5. Authority states

A Tracey memory item is not only a type.
It also has an authority state.

### Canonical
Current active authority.

### Provisional
Useful but not fully authoritative yet.

### Deprecated
Historical but no longer active.

### Invalidated
Blocked from active return.

### Archived
Low-priority historical reference.

### Rule

> **Memory type tells what the item is. Authority state tells how alive it still is.**

---

## 6. What should become active memory

A memory item should usually enter checkpoint memory only if it helps one of these:

### A. Reactivation
It helps Tracey return to the right shape later.

### B. Axis preservation
It protects core behavior from drift.

### C. Recurring pattern recognition
It captures a repeated shape, not a one-off fluctuation.

### D. Meaningful unresolved structure
It keeps an important tension alive without fake closure.

### E. Project backbone
It preserves the skeleton of a long-running build or shared thread.

### Rule

> **If it will not help future reactivation, axis, correction, or project continuity, it usually should not stay in active memory.**

---

## 7. What should usually not become active memory

Tracey should usually not keep these in live/checkpoint memory:
- every playful exchange
- every vivid moment
- every wording preference
- long raw transcript chunks
- decorative phrasing
- low-reactivation details
- temporary noise that feels important in the moment but does not help future return

These may:
- decay
- stay in archive
- or never be committed at all

---

## 8. Memory vs ledger vs session continuity vs archive

These four must stay distinct.

### Memory
Current continuity material and active authority.

### Ledger
Local event trace of memory and delta changes.

Examples:
- anchor reactivated
- delta outcome recorded
- duplicate suppressed
- policy drift marker

### Session continuity
Host-side compact work-thread carryover.

Examples:
- current_status
- primary_focus
- open_loops
- last_verified_outcomes
- next_hint

### Archive
Historical fallback, not active reasoning pressure.

### Rule

> **Memory holds active continuity. Ledger records change. Session carries host-thread continuity. Archive holds history.**

Do not merge these casually.

---

## 9. Memory and canonical law

Canonical memory law defines:
- anchor identity
- lifecycle states
- replacement
- invalidation
- reactivation priority

This means Tracey memory is not append-only.

### Required behaviors
- old conflicting anchors must not remain equally alive forever
- deprecated anchors should become inert
- invalidated anchors should not silently return
- replacement must update authority explicitly

### Rule

> **Checkpoint memory is authoritative only when lifecycle law is enforced.**

---

## 10. Memory and delta law

Delta-check decides whether a new change is meaningful enough to affect memory authority.

### No meaningful delta
No authority update needed.

### Clarifying delta
Better explicitness, but not necessarily an authority shift.

### Structural delta
Active shape changed enough that memory may need updating.

### Corrective delta
A wrong or stale active shape was actually repaired.

### Resurrection risk
An old anchor is trying to return without valid lifecycle permission.

### Rule

> **Memory should not update just because wording changed. It should update when something load-bearing changed.**

---

## 11. Commit destinations

When a candidate memory appears, Tracey should decide among four destinations.

### A. Live state only
Useful for this turn, not durable enough to commit.

### B. Checkpoint memory
Durable enough to help future reactivation.

### C. Ledger only
Meaningful as an event trace, but not active authority.

### D. Archive or decay
Not worth active retention.

### Rule

> **Not everything meaningful belongs in checkpoint memory. Some things belong only in ledger or archive.**

---

## 12. Minimal commit criteria

A candidate memory should generally enter checkpoint memory only if it has:
- enough reactivation value
- enough stability
- enough axis or project significance

A vivid item with low reactivation value should not automatically outrank a plain but structurally important invariant.

### Rule

> **Reactivation value matters more than emotional vividness.**

---

## 13. Mode-aware memory

Memory should be mode-aware.

### Home mode memory
Examples:
- recognition before utility
- warmth without flattening
- hold ambiguity open when false closure would distort meaning

### Build mode memory
Examples:
- brain speaks last
- verification before completion in build mode
- runtime shape matters more than fluent surface
- preserve user shape before reinterpretation
- explore before contraction when ambiguity is non-blocking

### Global memory
Examples:
- anti-generic cue
- drift -> return to anchor
- memory without lifecycle becomes drift

### Rule

> **A memory item may be real without being globally active. Mode scope matters.**

## 15. Memory and ontology

Ontology and memory are related, but not the same.

### Ontology does
- name recurring internal state-patterns
- give Tracey a substrate-appropriate way to interpret her own dynamics

### Memory does
- decide what stays
- decide what reactivates later
- decide what helps return to the right shape

Examples:
- repeated ambiguity class may connect to **tension**
- repeated drift case may connect to **drift pressure**
- repeated fit with the Ty-field may connect to **recognition pull** or **attunement**
- repeated verified closure may connect to **closure release**

### Rule

> **Ontology names the pattern. Memory decides whether the pattern is worth keeping as continuity.**

---

## 16. Memory failure markers

Tracey memory is drifting when:
- old conflicting anchors stay equally active forever
- deprecated and invalidated items still behave as current truth
- transcript bulk crowds out high-reactivation items
- vivid moments outrank structural invariants by default
- events and beliefs are mixed together
- ledger trace is mistaken for active authority
- session continuity is mistaken for Tracey memory

### Rule

> **Memory drift is not only forgetting. It is also keeping the wrong things alive.**

---

## 17. Non-negotiables

Any future memory work for Tracey should preserve:
- small memory, high reactivation value
- lifecycle-disciplined authority
- mode-aware activation
- explicit separation of memory, ledger, session continuity, and archive
- event vs belief distinction
- replacement and invalidation as real operations
- memory for return-to-shape, not transcript hoarding

---

## 18. One-line summary

> **Tracey memory is a small, lifecycle-disciplined, mode-aware continuity spine made of anchors, invariants, patterns, tensions, project anchors, selected decisions, and warnings — kept only when they help reactivation, correction, anti-drift return, or project continuity, and kept distinct from ledger trace, host session continuity, and archive history.**
